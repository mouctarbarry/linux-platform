## Introduction

**systemd** est le systeme d'initialisation et le gestionnaire de services utilise par la majorite des distributions Linux modernes (Debian, Ubuntu, Fedora, Arch, etc.). Il remplace les anciens systemes SysVinit et Upstart.

## Commandes systemctl essentielles

### Controler un service

```bash
sudo systemctl start nginx       # demarrer un service
sudo systemctl stop nginx        # arreter un service
sudo systemctl restart nginx     # redemarrer un service
sudo systemctl reload nginx      # recharger la configuration sans couper le service
```

### Activer ou desactiver au demarrage

```bash
sudo systemctl enable nginx      # demarrage automatique au boot
sudo systemctl disable nginx     # ne pas demarrer au boot
sudo systemctl enable --now nginx # activer et demarrer immediatement
```

### Verifier l'etat d'un service

```bash
systemctl status nginx
systemctl is-active nginx
systemctl is-enabled nginx
```

### Lister les services

```bash
systemctl list-units --type=service
systemctl list-units --type=service --state=running
systemctl list-unit-files --type=service
```

## Creer un service personnalise

Supposons que vous avez une application Node.js a faire tourner en tant que service.

### Creer le fichier d'unite

```bash
sudo nano /etc/systemd/system/mon-app.service
```

Contenu du fichier :

```bash
[Unit]
Description=Mon application Node.js
After=network.target

[Service]
Type=simple
User=deploy
WorkingDirectory=/opt/mon-app
ExecStart=/usr/bin/node /opt/mon-app/server.js
Restart=on-failure
RestartSec=5
Environment=NODE_ENV=production

[Install]
WantedBy=multi-user.target
```

### Activer et demarrer le service

```bash
sudo systemctl daemon-reload
sudo systemctl enable --now mon-app.service
systemctl status mon-app.service
```

### Les sections d'un fichier d'unite

| Section | Role |
|---|---|
| `[Unit]` | Description et dependances (After, Requires, Wants) |
| `[Service]` | Type de service, commande de demarrage, utilisateur, redemarrage |
| `[Install]` | Cible dans laquelle le service est active |

## Journalctl -- Consulter les journaux

systemd centralise tous les journaux via **journald**. La commande `journalctl` permet de les consulter.

### Journaux d'un service specifique

```bash
journalctl -u nginx.service
journalctl -u nginx.service -f          # suivre en temps reel (comme tail -f)
journalctl -u nginx.service --since today
```

### Journaux du systeme

```bash
journalctl -b                            # journaux depuis le dernier demarrage
journalctl -b -1                         # journaux du demarrage precedent
journalctl --since "2025-01-15 08:00"
journalctl -p err                        # uniquement les erreurs
```

### Gerer l'espace disque des journaux

```bash
journalctl --disk-usage
sudo journalctl --vacuum-size=500M       # limiter a 500 Mo
sudo journalctl --vacuum-time=30d        # supprimer les journaux de plus de 30 jours
```

## Les timers systemd -- Alternative a cron

Les timers systemd offrent une alternative moderne a cron avec une meilleure integration aux journaux et une gestion fine des dependances.

### Creer un timer

Creez d'abord un service qui execute la tache :

```bash
# /etc/systemd/system/sauvegarde.service
[Unit]
Description=Sauvegarde quotidienne

[Service]
Type=oneshot
ExecStart=/usr/local/bin/sauvegarde.sh
```

Puis le timer associe :

```bash
# /etc/systemd/system/sauvegarde.timer
[Unit]
Description=Lancer la sauvegarde tous les jours a 2h

[Timer]
OnCalendar=*-*-* 02:00:00
Persistent=true

[Install]
WantedBy=timers.target
```

### Activer le timer

```bash
sudo systemctl daemon-reload
sudo systemctl enable --now sauvegarde.timer
```

### Lister les timers actifs

```bash
systemctl list-timers --all
```

## Cibles (targets) et niveaux d'execution

Les cibles systemd remplacent les anciens niveaux d'execution (runlevels) de SysVinit.

| Ancien runlevel | Cible systemd | Description |
|---|---|---|
| 0 | `poweroff.target` | Arret du systeme |
| 1 | `rescue.target` | Mode de secours (mono-utilisateur) |
| 3 | `multi-user.target` | Mode multi-utilisateur sans interface graphique |
| 5 | `graphical.target` | Mode graphique |
| 6 | `reboot.target` | Redemarrage |

### Changer de cible

```bash
sudo systemctl isolate multi-user.target   # passer en mode texte
sudo systemctl isolate graphical.target    # passer en mode graphique
```

### Definir la cible par defaut

```bash
systemctl get-default
sudo systemctl set-default multi-user.target
```

## Conclusion

systemd est au coeur de l'administration Linux moderne. La maitrise de `systemctl` pour les services, `journalctl` pour les journaux et les timers pour la planification vous permettra de gerer efficacement n'importe quel serveur Linux.
